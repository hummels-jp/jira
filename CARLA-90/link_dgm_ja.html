リンク 1：DGMデータソースに基づく

コアパス： 武漢DGMデータ (OpenDrive) → RoadRunner → CARLA

詳細な処理フロー：

  • データ前処理： 武漢側のDGMから変換されたOpenDRIVEファイルのバージョン検証とトポロジーチェックを行います。
  • RoadRunnerモデリング： OpenDRIVEファイルをRoadRunnerにインポートし、ジオメトリのインスタンス化を行い、路面材質や物理属性を補完します。
  • プラグインエクスポート： RoadRunnerのCARLAプラグインを利用してFBXファイルと対応するxodrファイルを出力し、座標合わせを確認します。

技術検証 (QA)

Q: NOA_CITYWAY_V3.0.xodr ファイルの分析結論は？

A: このファイルは路網構築の基礎要素（トポロジー接続、幾何軌跡、車線構造、初期の交通標識）を備えていますが、以下の制限に注意が必要です。

  • トポロジーの完全性： `link` 関係は良好ですが、交差点（Junction）内部のすべての接続に正確な `laneLink` が提供されているか再確認が必要です。
  • 信号機のセマンティクス： `type="294"` (停止線) や `type="293"` (横断歩道) の信号情報は識別されましたが、コントローラー定義が欠けています。動的な信号機シミュレーションにはController連携が必要です。
  • 物理属性： Surface定義は現在空であり、摩擦係数などの動力学シミュレーションに不可欠なパラメータが不足しています。
  • 今後の検証建議： odr-viewer で孤立ノードをチェックし、AutowareマップローダーでLanelet2変換テストを実施し、セマンティック属性が正しく保持されているか検証することを推奨します。

Q: DGMデータの変換フローにおいて、元のOpenDriveデータは様々なツールでどのように表示されますか？

A: 元のDGMから変換されたOpenDriveデータは、`odrViewer` や `RoadRunner` 内で高度に構造化されたベクトルグリッドとして表示され、以下の特徴を持ちます。

  • 直感的なトポロジーネットワーク： 各Road IDは一貫した道路として解析され、viewer内で都市路網の骨格として可視化されます。
  • 幾何学的忠実度： RoadNetworkは層状に表示され、精緻な参照線により武漢の現実の路網のカーブや交差形態を再現しています。
  • 構造的一貫性： OpenDRIVE仕様に厳密に従っており、データ構造の整合性が検証されています。

Q: OpenDriveデータをCARLAで使用可能なFBXシーンに変換する方法は？

A: RoadRunnerの CreateScene ワークフローにより、効率的な変換が可能です。

  • インポート： OpenDriveファイル (NOA_CITYWAY_V3.0.xodr) を直接RoadRunnerシーン編集画面へドラッグします。
  • 作成とインスタンス化： CreateScene コマンドによりデータをインスタンス化し、道路メッシュや材質を自動生成します。
  • 調整と確認： RoadRunner画面でNOA_CITYWAY_V3.0.xodrの全容を確認し、ネットワークの断裂や交差点ロジックを確認します。
  • エクスポート： CARLAエクスポートプラグインを使用してFBXを出力し、xodrファイルと合わせてCARLAへインポートします。

Q4: Carla UE Editor 環境で make import コマンドを実行し、FBX 等のリソースを Carla Editor にインポートする方法は？

A: Unreal Engine Editor 環境下で、Makefile または手動インポートフローを使用して処理された FBX およびアセットを Carla にデプロイします。

  • Content Browser 構造： 図の Content Browser に示すように、マップリソースは Content > map_package > Maps > kawasaki パス下に編成されています。
  • アセット統合： インポート後、モデルは StaticMeshActor (kawasaki_Roads_Road_Layer0 等) として認識され、World Outliner に正しくマウントされます。
  • コンパイルとビルド： Editor 画面で `Build` オプションを介して Mesh Distance Fields をコンパイルします（右下に表示）。これは路面の衝突検知やセンサーシミュレーション（特にLiDAR）に不可欠です。
  • デプロイ検証： インポート後、Viewport 上で kawasaki マップが完全にロードされているか、すべての材質（LaneMarking_Yellow1 等）が期待通りにレンダリングされているか確認し、シミュレーション環境と現実の物理形態の一致を確保します。
