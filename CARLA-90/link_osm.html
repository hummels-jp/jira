链路 2：基于 OSM 数据源

核心路径： OSM → SUMO → OpenDrive → RoadRunner → CARLA

详细处理流程：

  • 数据格式转换： 利用 SUMO 的 `netconvert` 工具将 OSM 网络数据转换为 OpenDRIVE 格式。
  • 拓扑重构： 在 SUMO 转换过程中重点处理交叉口逻辑及车道连接，确保逻辑连通性。
  • 场景实例化： 同步进入 RoadRunner 流程，完成场景的物理视觉构建。

技术验证 (QA)

Q1: 如何从 OpenStreetMap 网站导出所需区域的 OSM 地图数据？

A: 获取 OSM 原始数据是进行仿真场景构建的第一步，可以通过以下标准操作流程完成：

  • 区域定位： 访问 OpenStreetMap 官网，在搜索栏或地图拖动中定位至目标区域。
  • 数据导出： 点击顶部的“导出”按钮，手动调整界面的选框 (Bounding Box)，确保涵盖完整范围。
  • 获取格式： 下载后缀为 .osm 的 XML 格式文件。

Q2: 通过 SUMO 的 netconvert 工具将 OSM 数据转换为 OpenDrive 数据的处理结果如何？

A: 通过 SUMO 的转换工具，可以成功将原始的 OSM 语义网络转化为仿真器可读的 OpenDrive 格式，对于 kawasaki 和 negishi 区域的转换结果清晰完整。

Q3: 通过 RoadRunner 将生成的 OpenDrive 数据创建为 Scene 并导出为 FBX 中间格式的逻辑是什么？

A: 通过 RoadRunner 的 Scene 编辑环境，可以将 OpenDrive 逻辑拓扑转化为 CARLA 可用的高性能仿真资产：

  • 可视化编辑与实例化： 如截图所示，在 RoadRunner 的场景编辑界面中导入 negishi.xodr，系统会自动将其渲染为 3D 路网逻辑图，便于直观检查拓扑是否完整，是否存在“路口断裂”或“车道连接错误”等常见问题。
  • 构建 Scene 容器： 使用 `CreateScene` 命令将 OpenDrive 定义的逻辑拓扑与 RoadRunner 的 3D 资产库（如标志、路标、地面模型）进行绑定，形成一个完整的仿真 Scene。
  • FBX 导出策略： 导出 FBX 是关键的一步。FBX 不仅仅是一个几何网格，还包含了材质信息与坐标原点偏移 (Transform) 信息。导出时选择 CARLA 导出插件，确保导出的 xodr 与 FBX 在坐标系和原点对齐上达到一致。
  • CARLA 衔接： 最终导入 Carla 引擎时，CARLA 地图引擎将直接读取该 xodr 文件作为路网逻辑引导，使用 FBX 模型作为视觉展示，从而实现逻辑与视觉的完美契合。
  • 补充说明（闭环问题）： 由于 OSM 原始数据源的道路闭合性（Road Closure）较差，在 RoadRunner 中使用 `CreateScene` 时，部分区域的道路末端可能无法自动对齐，从而未能形成逻辑闭环。这通常会导致仿真中的车辆在末端无法寻路，后续需在 RoadRunner 环境内进行手动缝合（Stitching）操作。

Q5: OpenDrive 数据支持路径计算吗？

A: 是的，OpenDrive 格式设计之初就明确支持路径规划与计算。它通过以下机制支撑这一核心功能：

  • 路网拓扑结构 (Road Network Topology): OpenDrive 定义了严密的 <road>、<laneSection> 及 <lane> 层次，并通过 predecessor/successor 逻辑将所有车道串联为完整的拓扑图。
  • 几何路径参数化: 每一段道路均通过解析几何（如 line, arc, spiral（回旋曲线）、poly3 等）定义中心线，路径规划器可直接计算出车辆在该曲线上的空间坐标与航向角。
  • 连接点逻辑 (Junction Logic): 通过 <junction> 及 <connection> 定义了复杂的路口通行规则与车道切换路径，为全局路径规划提供了底层的通行逻辑支持。
  • 高精路网查询: 路径规划算法（如 A*, Dijkstra）可直接读取 OpenDrive 中的路网图结构，从而实现从起点到终点的车道级路径计算。

Q6: Autoware 为什么使用 Lanelet2 地图格式，而不直接使用 OpenDrive 地图格式？

A: Autoware 选用 Lanelet2 而非 OpenDrive 作为核心地图格式，主要基于其更适合“实时感知与动态规划”的设计理念：

  • 计算效率与轻量化： OpenDrive 面向仿真，包含大量复杂的几何逻辑与 3D 属性描述，解析负载高；Lanelet2 基于点序列拓扑（Primitives），不仅存储空间极小，且 Autoware 的规划器可直接在内存中对其进行高效的动态寻路，响应速度快。
  • 动态通行逻辑的可编程性： Lanelet2 基于“Lanelet”元数据定义（如路权、限速、交通标志），非常易于与高精感知模块交互；Autoware 可直接通过简单的几何交集运算动态判断车辆是否位于有效的 Lanelet 区域内。
  • 语义层级与空间索引： Lanelet2 内置了成熟的 R-tree 空间索引，Autoware 在执行感知匹配（Localization）时，无需遍历全图，即可在毫秒级内锁定车辆所在的当前 Lanelet 及其关联的临近路段。
  • 开放性与实时修正： 仿真（OpenDrive）注重物理一致性，而实车（Autoware）注重场景语义的动态更新；Lanelet2 的模块化架构（Point-Lanelet-Area）支持在运行时对地图进行热修复（如临时道路封闭、绕行标记），无需重新加载庞大的物理仿真模型。
