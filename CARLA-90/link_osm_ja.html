リンク 2：OSMデータソースに基づく

コアパス： OSM → SUMO → OpenDrive → RoadRunner → CARLA

詳細な処理フロー：

  • データ形式変換： SUMOの `netconvert` ツールを使用して、OSMネットワークデータをOpenDRIVE形式に変換します。
  • トポロジー再構築： SUMO変換過程で交差点ロジックや車線接続を重点的に処理し、論理的な接続性を確保します。
  • シーンインスタンス化： RoadRunnerフローへ同期させ、物理的な視覚構築を完了します。

技術検証 (QA)

Q: OpenStreetMap サイトから必要なエリアの OSM 地図データをエクスポートする方法は？

A: シミュレーションシーン構築の第一歩として、以下の標準手順で OSM データを取得します。

  • エリア指定： OpenStreetMap公式サイトにアクセスし、検索バーや地図のドラッグでターゲットエリアを特定します。
  • データエクスポート： 上部の「エクスポート」ボタンをクリックし、範囲選択 (Bounding Box) を手動調整して必要な範囲をカバーします。
  • 形式の取得： .osm という拡張子の XML ファイルをダウンロードします。

Q: SUMO の netconvert ツールによる OSM データから OpenDrive への変換結果は？

A: SUMO 変換ツールを用いることで、原始的な OSM セマンティックネットワークをシミュレーターが読み取り可能な OpenDrive 形式に変換可能です。川崎や根岸エリアの変換結果は明確で完全です。

Q: RoadRunner を使用して生成された OpenDrive データを Scene として作成し、FBX 中間形式としてエクスポートするロジックは？

A: RoadRunner のシーン編集環境を通じて、OpenDrive の論理トポロジーを CARLA が利用可能な高性能なシミュレーションアセットへ変換します。

  • 視覚的編集とインスタンス化： negishi.xodr をインポートすると、3D 路網ロジック図としてレンダリングされ、トポロジーの欠落や接続エラーを直感的に確認できます。
  • Scene コンテナの構築： `CreateScene` コマンドにより、論理トポロジーとアセットライブラリを結合し、完全なシミュレーション Scene を形成します。
  • FBX エクスポート戦略： FBX 変換は重要です。座標原点のオフセットや材質情報を含み、CARLA プラグインを使用して正しい軸系で出力します。
  • CARLA への連携： CARLA エンジンは xodr ファイルを論理ガイドとして読み込み、FBX モデルを視覚要素として表示し、両者の整合性を保ちます。
  • 補足（クローズドループの課題）： OSM ソースデータの道路の閉塞性 (Road Closure) が低いため、`CreateScene` を使用した際に道路端が自動で接続されず、論理クローズドループを形成できない場合があります。これに対しては RoadRunner 内での手動縫合 (Stitching) が必要となります。

Q4: Carla UE Editor 環境で make import コマンドを実行し、FBX 等のリソースを Carla Editor にインポートする方法は？

A: Unreal Engine Editor 環境下で、Makefile または手動インポートフローを使用して処理された FBX およびアセットを Carla にデプロイします。

  • Content Browser 構造： 図の Content Browser に示すように、マップリソースは Content > map_package > Maps > kawasaki パス下に編成されています。
  • アセット統合： インポート後、モデルは StaticMeshActor (kawasaki_Roads_Road_Layer0 等) として認識され、World Outliner に正しくマウントされます。
  • コンパイルとビルド： Editor 画面で `Build` オプションを介して Mesh Distance Fields をコンパイルします（右下に表示）。これは路面の衝突検知やセンサーシミュレーション（特にLiDAR）に不可欠です。
  • デプロイ検証： インポート後、Viewport 上で kawasaki マップが完全にロードされているか、すべての材質（LaneMarking_Yellow1 等）が期待通りにレンダリングされているか確認し、シミュレーション環境と現実の物理形態の一致を確保します。

Q5: OpenDrive データは経路計算をサポートしていますか？

A: はい、OpenDrive 形式は設計当初より、経路計画と計算を明示的にサポートしています。以下のメカニズムによってこのコア機能を支えています。

  • 道路ネットワークトポロジー (Road Network Topology): OpenDrive は厳密な <road>、<laneSection>、および <lane> 階層を定義し、predecessor/successor ロジックを通じてすべての車線を完全なトポロジーグラフに接続します。
  • 幾何学的経路パラメータ化: 各道路セグメントは、解析幾何学（line, arc, spiral（クロソイド曲線）、poly3 等）によって中心線を定義しており、経路計画器は車両の当該曲線上の空間座標と方位角を直接計算できます。
  • 接続点ロジック (Junction Logic): <junction> および <connection> を介して、複雑な交差点の通行ルールと車線変更経路を定義し、グローバル経路計画に底層の通行ロジックを提供します。
  • 高精度路網クエリ: 経路計画アルゴリズム（A*, Dijkstra 等）は、OpenDrive 内の路網グラフ構造を直接読み取り、始点から終点までの車線レベルでの経路計算を実現できます。

Q6: Autoware が OpenDrive 地図形式を直接使用せず、Lanelet2 地図形式を採用している理由は？

A: Autoware が OpenDrive ではなく Lanelet2 を核心地図形式に選定した理由は、「リアルタイム認識と動的計画」に適した設計思想に基づいています。

  • 計算効率と軽量化： OpenDrive はシミュレーション向けであり、複雑な幾何学的ロジックや 3D 属性定義を多く含むため解析負荷が高いです。対して Lanelet2 は点シーアンストポロジー（Primitives）に基づいており、ストレージサイズが非常に小さく、Autoware の計画器はメモリ内で直接効率的に動的経路探索を行うことができ、レスポンスが高速です。
  • 動的通行ロジックのプログラム可能性： Lanelet2 は「Lanelet」メタデータ定義（優先権、制限速度、交通標識等）に基づいており、高度な認識モジュールとの対話が容易です。Autoware は簡単な幾何学演算を介して、車両が有効な Lanelet エリア内にいるかを動的に判断できます。
  • セマンティック階層と空間インデックス： Lanelet2 は洗練された R-tree 空間インデックスを内蔵しており、Autoware が認識マッチング（Localization）を実行する際、全地図を探索する必要がなく、ミリ秒単位で現在の Lanelet と関連する周辺道路を特定できます。
  • 開放性とリアルタイム修正： シミュレーション (OpenDrive) は物理的一致性を重視しますが、実車 (Autoware) はシーンセマンティクスの動的更新を重視します。Lanelet2 のモジュール構造 (Point-Lanelet-Area) は、稼働中に地図をホットフィックス（一時的な道路封鎖や迂回マーキング等）することをサポートしており、巨大な物理シミュレーションモデルを再ロードする必要はありません。
